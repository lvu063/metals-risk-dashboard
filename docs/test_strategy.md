## 1. Purpose and Scope

This document defines the test strategy, test cases, and sign-off process for Phase 1 of the Metals Trading Risk Dashboard. It ensures the system meets all functional requirements (FR-01–FR-24) and non-functional requirements (NFR-01–NFR-06) defined in the BRD before go-live.

### 1.1 What is being tested

| Component | Included | Notes |
|---|---|---|
| ETL pipeline (Python) | ✅ | All four CSV → DB flows |
| SQL analytics queries | ✅ | PnL Explained, risk summary |
| Power BI dashboard | ✅ | All four pages |
| Power Automate flows | ✅ | Flows 1–3; Flow 4 (override) in regression |
| Excel workbook | ✅ | Formulas, VBA macros |
| Database schema | ✅ | Constraints, indexes, referential integrity |
| Endur export parsing | ✅ | Edge cases and known quirks |
| Security / RLS | ✅ | Role-based access for each desk |

### 1.2 What is NOT being tested (Phase 1)

- Realised PnL calculation (Phase 2)
- VaR engine (Phase 3)
- Direct Endur Oracle DB connection (Phase 3)
- Options greeks (Phase 3)
- Non-USD currency conversion (Phase 2)

## 2. Test Approach

Testing is structured across four levels, executed in sequence:

```
Level 1          Level 2          Level 3          Level 4
Unit Tests    Integration     System / E2E         UAT
──────────    ───────────     ────────────     ────────────
Automated     Automated +     Manual + auto    Stakeholder
Developer-    BA-led          BA + QA          sign-off
owned                                          required
```

### 2.1 Level 1 — Unit Testing (Developer-owned)

**Scope:** Individual Python functions, SQL query logic, DAX measure outputs  
**Who:** Developer / BA with SQL/Python access  
**When:** During development; re-run on every code change  
**Tools:** Python `unittest`, SQLite test DB, manual DAX validation in Power BI Desktop

### 2.2 Level 2 — Integration Testing (BA-led)

**Scope:** End-to-end flow from CSV input → DB → dashboard visual  
**Who:** BA + Developer  
**When:** After each component is built; before UAT  
**Tools:** Prepared test data sets (normal, edge case, error)

### 2.3 Level 3 — System Testing (BA + QA)

**Scope:** Full pipeline with production-volume data; performance; security  
**Who:** BA, QA analyst, IT  
**When:** Two weeks before UAT  
**Tools:** 1,000-row stress data set, Power BI service (not Desktop)

### 2.4 Level 4 — UAT (Stakeholder sign-off)

**Scope:** Business validation by desk users and risk team  
**Who:** Front Office traders, Market Risk manager, Middle Office lead  
**When:** Final two weeks before go-live  
**Tools:** UAT environment mirroring production; test scripts below

## 3. Test Data Strategy

Three test data sets are used, each serving a different purpose:

### Dataset A — Happy Path (300 trades, 8 metals, full year 2024)
Generated by `generate_mock_data.py --rows 300 --seed 42`

Used for: Level 1 & 2 tests, baseline UAT  
All statuses represented · No missing fields · All symbols valid

### Dataset B — Edge Cases (50 targeted records)
Hand-crafted CSV to trigger known failure modes:

| Record | Scenario | Expected behaviour |
|---|---|---|
| B-001 | Cancelled trade | Excluded from positions and PnL |
| B-002 | Null settlement date | Defaults to trade_date + 2, warning logged |
| B-003 | Zero quantity | Rejected — fails `positive_number` validator |
| B-004 | Unknown symbol `XPB` | Rejected — fails `valid_symbol` validator |
| B-005 | Negative notional | Accepted (short trade), sign preserved |
| B-006 | Settlement < trade date | Accepted with warning in rejection_log |
| B-007 | Duplicate trade_id | Second occurrence rejected as duplicate |
| B-008 | Symbol `XAU COMDTY` (Endur format) | Stripped to `XAU` in transform |
| B-009 | Multi-leg swap (2 rows, same tran) | Aggregated into single trade record |
| B-010 | Empty price row | Rejected — fails `positive_number` on spot_price |

### Dataset C — Volume / Performance (5,000 trades, full year)
Generated by `generate_mock_data.py --rows 5000 --seed 99`

Used for: NFR-01 (ETL SLA), NFR-02 (dashboard load time)  
Not used for functional testing — volume only

## 6. Defect Management

### 6.1 Severity Classification

| Severity | Definition | Example | Resolution SLA |
|---|---|---|---|
| **P1 — Critical** | System unusable; data corruption risk; PnL materially incorrect | ETL loads wrong notional values; dashboard blank for all users | 4 hours |
| **P2 — High** | Core function broken; affects multiple users; workaround unavailable | Drill-through navigation broken; RLS not filtering | 1 business day |
| **P3 — Medium** | Feature impaired; workaround available | Concentration flag missing; VBA macro fails on one machine | 3 business days |
| **P4 — Low** | Cosmetic, minor formatting, or nice-to-have | Wrong colour on a chart; column width in Excel | Next sprint |

### 6.2 Defect Triage Process

```
Tester finds defect
       │
       ▼
Log in JIRA (project: MTD)
  · Summary
  · Severity (P1–P4)
  · Steps to reproduce
  · Expected vs actual result
  · Screenshot / log file attached
  · BRD requirement reference (FR-XX)
       │
       ▼
BA triages within 4h (P1) / 1 day (P2–P4)
  · Confirm severity
  · Assign to developer
  · Update test case status → FAIL
       │
       ▼
Developer fixes & deploys to test environment
       │
       ▼
BA / Tester re-runs failed test case
  · PASS → close defect, mark TC as PASS
  · FAIL → reopen defect, escalate if >2 reraises
```

### 6.3 Go/No-Go Criteria

The system may proceed to go-live **only when** all of the following are met:

| Criterion | Threshold |
|---|---|
| P1 defects open | 0 |
| P2 defects open | 0 |
| P3 defects open | ≤ 2 (with documented workarounds) |
| Acceptance criteria (AC-01–AC-06) | All 6 passed |
| NFR test cases | All 6 passed |
| Stakeholder sign-off received | All 4 signatories |
